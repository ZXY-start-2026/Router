# 多模型路由聊天系统需求

## 0. 范围

实现一个支持以下能力的多轮对话系统：

- 消息级自动模型路由
- 单条消息临时手动指定模型
- 每条新消息强制执行 360 搜索
- 修改历史用户消息时创建对话分支
- 同一用户消息支持多个助手回答版本
- 分支级、版本化自然语言备忘录
- 会话角色/人格定制
- 左右分栏聊天界面
- 固定接入 3 个候选大模型

不得物理删除或覆盖以下历史数据：

- 对话分支
- 助手回答版本
- 备忘录版本
- 路由快照
- 搜索快照
- 模型生成尝试记录

---

## 1. 已确定的产品规则

```yaml
候选模型数量: 3

路由:
  粒度: 消息级
  准确率权重: 0.70
  成本权重: 0.30
  准确率含义: 当前问题由该模型答对的预测概率
  手动指定模型作用范围: 仅当前消息

搜索:
  服务商: 360搜索
  每条新用户消息强制搜索: true
  最大结果数: 5
  传给模型的字段:
    - 标题
    - 摘要
  抓取网页正文: false
  回答中显示引用编号: false
  回答中显示来源链接: false
  重新生成时复用搜索快照: true

备忘录:
  格式: 自然语言
  首次触发总完整轮数_n: 10
  每次更新后保留最近原始轮数_k: 5
  批量更新轮数_m: 5
  参数可配置: true
  用户维护文本受保护: true
  所有变更均创建版本: true
  旧版本允许删除: false

历史修改:
  行为: 创建新分支
  保留原分支: true

回答版本:
  每条用户消息最多一个当前生效回答: true
  成功重新生成后自动设为当前版本: true

自动失败降级:
  暂时性错误同模型自动重试一次: true
  按本轮路由排序继续尝试其他模型: true
```

## 2. 核心术语

### 完整轮次

一个完整轮次定义为：

```text
一条用户消息 + 一条成功完成且当前生效的助手回答
```

以下情况不计入完整轮次：

- 模型调用失败
- 用户停止生成
- 只生成了部分回答
- 用户消息仍处于生成中

同一用户消息存在多个回答版本或生成尝试时，只要最终有一个成功回答成为当前生效版本，该用户消息只计为 1 个完整轮次。

### 用户消息节点

一条用户消息以及与其关联的：

- 一个搜索快照
- 零个或多个生成任务
- 零个或多个助手回答版本
- 最多一个当前生效回答版本

### 对话分支

一条线性的有效对话路径。多个分支可以共享分叉点之前的不可变历史，在分叉点之后独立发展。

### 搜索快照

为一条用户消息生成的不可变搜索数据。同一用户消息的所有重新生成操作复用该快照。

### 备忘录版本

属于某个对话分支的版本化上下文摘要。

## 3. 页面要求

### 页面布局

- 左侧：会话列表
- 右侧：当前会话的当前分支内容

### 左侧会话列表

- 首次加载最近 20 个会话
- 按 `updated_at DESC` 排序
- 支持滚动加载更多
- 使用基于游标的稳定分页，不使用页码分页
- 每个会话至少展示：
  - 会话标题
  - 最近消息摘要
  - 最近更新时间
  - 当前生成状态

### 对话顶部区域

至少提供：

- 会话标题
- 当前角色入口
- 当前备忘录入口
- 存在分支时的分支指示器和切换入口

### 输入区域

模型选择器提供：

```text
自动路由
仅本条消息使用模型 A
仅本条消息使用模型 B
仅本条消息使用模型 C
```

无论当前消息生成成功或失败，手动模型选择在本次生成任务结束后都恢复为自动路由。

### 助手消息元信息

默认紧凑展示：

```text
<模型名称> · <自动路由 | 自动降级 | 用户指定>
```

展开详情可展示：

- 初始路由模型
- 最终实际回答模型
- 候选模型评分
- 预测准确率
- 预测/实际 Token 数
- 预测/实际成本
- 路由策略版本
- 搜索状态
- 生成尝试链路
- 失败与降级原因

## 4. 新消息处理流程

### 4.1 通用流程

每条新用户消息依次执行：

1. 确定当前会话
2. 确定当前活动分支
3. 确定当前生效角色版本
4. 确定当前生效备忘录版本
5. 确定该分支中每条历史用户消息的当前生效回答
6. 保存新用户消息
7. 创建搜索快照
8. 创建不可变的生成上下文快照
9. 根据自动路由或用户手动指定确定模型
10. 执行模型生成
11. 保存生成尝试和回答版本
12. 成功回答自动设为当前生效版本
13. 更新完整轮次数
14. 满足条件时更新备忘录

### 4.2 生成上下文快照

同一个生成任务中的所有模型尝试必须使用完全一致的上下文快照。

上下文按以下顺序组成：

1. 系统规则
2. 当前角色版本
3. 用户维护的受保护备忘录文本
4. 系统生成的备忘录摘要
5. 未进入备忘录覆盖范围的有效原始对话
6. 本轮搜索结果标题/摘要，或明确的搜索失败状态
7. 当前用户消息

不得为了迁就上下文较短的降级模型而静默删除有效历史。无法接收完整必需上下文的模型应在候选筛选阶段被排除或跳过。

## 5. 360 搜索

### 5.1 触发规则

每条新用户消息都必须尝试执行 360 搜索，包括：

- “继续”
- “再解释一下”
- “改写上面的内容”
- 问候语
- 依赖上下文的指代问题

搜索查询词可以结合：

- 当前用户消息
- 当前生效备忘录
- 最近原始对话
- 当前角色的领域信息

当用户消息不能独立作为搜索词时，系统必须结合上下文将其改写为可独立理解的搜索查询。

### 5.2 结果规则

- 最多取前 5 条有效结果
- 去除重复或高度相似结果
- 去重后不足 5 条时，使用实际数量
- 只将标题和摘要传给模型
- 不抓取网页正文
- 系统内部保留 URL 用于追溯，但不在回答中展示

### 5.3 搜索状态

```yaml
SUCCESS_WITH_RESULTS
SUCCESS_NO_VALID_RESULTS
FAILED
TIMEOUT
```

处理规则：

- `SUCCESS_WITH_RESULTS`：将结果传给模型
- 其他状态：不提供搜索内容，但继续生成回答
- 必须明确告诉模型，本轮没有获得可用的联网搜索信息
- 页面必须明确标记本回答未成功使用本轮联网搜索

对于依赖实时信息的问题，搜索失败后模型不得伪装为已经完成实时核验。

### 5.4 搜索快照复用

以下场景必须复用原搜索快照：

- 使用原模型重新生成
- 重新自动路由生成
- 临时指定模型重新生成
- 模型调用失败后的重试
- 同一生成任务内的自动降级
- 同一用户消息更换回答生成方式

以下场景创建新搜索快照：

- 创建新的用户消息
- 修改历史用户消息后创建新分支和新用户消息节点

## 6. 自动路由

### 6.1 候选模型硬性筛选

在评分前，对 3 个模型分别检查：

- 是否启用并已配置
- 当前是否可用
- 是否支持完整上下文长度
- 是否支持本轮所需输入/输出能力
- 是否支持本轮所需请求参数

硬性条件优先于路由得分。

### 6.2 每模型预测输出

对每个满足硬性条件的模型输出：

```yaml
预测准确率:
  含义: 当前问题由该模型答对的概率

预测输出Token数:
  含义: 该模型针对当前请求预计生成的Token数量
```

输入 Token 数必须使用该候选模型自己的分词器，对完整生成上下文分别计算。

### 6.3 预计成本

```text
预计成本 =
  预计输入Token数 × 模型输入Token单价
  + 预测输出Token数 × 模型输出Token单价
```

必须保存本次计算使用的价格版本。

### 6.4 成本得分

成本需要转换为“越高越优”的分数。默认规则：

```text
成本得分(模型) =
  所有合格模型中的最低预计成本
  ÷ 当前模型预计成本
```

最便宜的合格模型成本得分为 1。

### 6.5 综合路由得分

```text
综合路由得分 =
  0.70 × 预测准确率
  + 0.30 × 成本得分
```

所有合格模型按综合路由得分降序排列。

当前未定义最低预测准确率门槛。

### 6.6 路由快照

每次自动路由必须保存不可变快照，包括：

- 路由策略/预测模型版本
- 候选模型集合
- 每个模型是否合格
- 不合格原因
- 每个合格模型的预测准确率
- 预测输入 Token 数
- 预测输出 Token 数
- 预计成本
- 成本得分
- 综合路由得分
- 排名顺序
- 路由耗时
- 价格版本

同一个生成任务中的自动降级必须沿用该排序，不重新执行路由。

---

## 7. 手动指定模型

用户手动指定模型时：

- 仅对当前消息或当前重新生成任务生效
- 仍然执行强制搜索
- 跳过自动路由
- 直接调用用户指定模型
- 选择方式记录为 `USER_SELECTED`
- 本轮结束后恢复自动路由

若指定模型调用失败：

- 不自动切换模型
- 保留失败尝试
- 保留搜索快照
- 保留用户消息
- 提示用户选择：
  - 重试同一模型
  - 选择其他模型
  - 改用自动路由

成功重试或重新选择模型后，为原用户消息生成新的回答版本，不重复创建用户消息，也不创建新分支。

---

## 8. 模型调用失败处理

### 8.1 暂时性错误

包括：

- 超时
- 服务不可用
- 限流
- 连接失败
- 服务端异常

自动路由模式处理：

1. 同一模型自动重试一次
2. 重试失败后，尝试路由排序中的下一个合格模型
3. 每个后续模型也允许一次重试
4. 三个模型全部耗尽后结束

手动指定模式：

- 不自动降级
- 由用户选择后续操作

### 8.2 模型特定的确定性错误

包括：

- 当前模型上下文容量不足
- 当前模型不支持所需功能
- 当前模型不支持某项请求参数
- 当前模型参数格式不兼容
- 当前模型被停用或配置不可用

处理：

- 不重试同一模型
- 自动模式下跳过当前模型，尝试下一个兼容模型
- 能在调用前检测时，应提前过滤

### 8.3 全局确定性错误

包括：

- 输入内容违规
- 全局请求参数非法
- 必要请求数据缺失
- 上下文组装异常
- 该请求无法被任何模型合法执行

处理：

- 立即终止本轮生成
- 不重试
- 不自动降级
- 保存失败状态和具体原因

输入内容违规始终是全局错误，禁止通过切换模型绕过。

### 8.4 三个模型全部失败

- 保留用户消息
- 保留搜索快照
- 不创建生效助手回答
- 不增加完整轮次数
- 不触发备忘录更新
- 允许用户：
  - 重试原路由模型
  - 重新自动路由
  - 临时指定模型

### 8.5 生成尝试记录

每个生成任务保存有序尝试记录：

```yaml
attempt_index
model_id
started_at
ended_at
status
error_category
error_code
error_message
retry_of_attempt_id
input_tokens_if_known
output_tokens_if_known
cost_if_charged
```

自动降级中的多次模型尝试属于同一个回答版本生成任务，不是多个回答版本。

## 9. 实际成本

模型成功返回后计算：

```text
实际成本 =
  实际输入Token数 × 输入Token单价
  + 实际输出Token数 × 输出Token单价
```

必须保存：

- 预测输入/输出 Token 数
- 实际输入/输出 Token 数
- 预计成本
- 实际成本
- 预测误差
- 价格版本

初始路由模型和最终实际回答模型可以不同。

---

## 10. 回答版本与重新生成

### 10.1 重新生成菜单

对已有回答提供：

1. 使用原模型重新生成
2. 重新自动路由
3. 临时指定模型重新生成

三种方式都复用原搜索快照。

### 10.2 使用原模型重新生成

- 使用当前所选回答版本实际生成时的模型
- 跳过路由
- 复用搜索快照
- 创建新回答版本

### 10.3 重新自动路由

- 复用原用户消息和原搜索快照
- 使用该消息位置当时有效的历史上下文重新构建请求
- 使用当前路由预测、模型状态和价格重新路由
- 创建新的路由快照
- 创建新的回答版本

### 10.4 临时指定模型重新生成

- 用户临时选择一个模型
- 跳过路由
- 复用搜索快照
- 仅对本次重新生成生效
- 创建新的回答版本

### 10.5 当前生效回答

- 每条用户消息最多一个成功且当前生效的助手回答
- 新回答成功生成后自动成为当前生效版本
- 旧版本继续保存并可查看
- 失败或停止的重新生成不得替换当前生效版本
- 后续上下文只使用当前生效回答

### 10.6 切换回答版本

仅查看旧版本不影响上下文。

点击“设为当前版本”时：

- 如果该回答之后没有后续消息：直接切换当前版本
- 如果该回答之后已有后续消息：从该回答版本处创建新分支
- 不复制原分支中的后续消息

如果该历史回答已经进入某个备忘录版本，新分支只能继承覆盖范围严格结束于该回答位置之前的备忘录版本。

## 11. 修改历史用户消息

编辑历史用户消息时创建新分支。

示例：

```text
原分支:
U1 -> A1 -> U2 -> A2 -> U3 -> A3

将 U2 修改为 U2':

原分支保留:
U1 -> A1 -> U2 -> A2 -> U3 -> A3

新活动分支:
U1 -> A1 -> U2' -> A2'
```

规则：

- 保留原分支
- 分叉点之前共享不可变历史
- 为修改后的内容创建新用户消息节点
- 重新执行搜索
- 默认重新自动路由，也允许用户仅本条指定模型
- 新分支创建后自动成为活动分支
- 后续消息只追加到活动分支
- 用户可以切换分支

---

## 12. 会话备忘录

### 12.1 结构

每个分支独立维护备忘录版本。

每个备忘录版本包含两个逻辑区域：

```yaml
用户保护区:
  用户可编辑: true
  系统可编辑: false

系统摘要区:
  系统自动生成: true
  允许系统整理和压缩: true
```

页面可以放在同一个备忘录面板中，但必须明确区分两个区域。

### 12.2 上下文冲突优先级

模型解释上下文时使用：

```text
当前用户消息
> 最近原始对话
> 用户保护区
> 系统摘要区
```

如果后续原始对话与用户保护区冲突：

- 系统必须标记冲突
- 系统不得自动修改用户保护区
- 由用户决定是否编辑

### 12.3 默认参数

```yaml
n: 10  # 首次生成备忘录所需完整轮数
k: 5   # 每次更新后保留的最近原始轮数
m: 5   # 每累计多少个新完整轮次进行一次更新
```

三项均可配置。

### 12.4 首次生成

第 10 个完整轮次成功后：

- 总结第 1-5 轮
- 第 6-10 轮保留原文
- 创建新的备忘录版本

第 10 轮之前，所有历史均保留原文。

### 12.5 批量更新

首次备忘录生成后，每新增 5 个完整轮次更新一次。

示例：

| 完整轮数 | 备忘录覆盖 | 原始对话 |
|---:|---|---|
| 10 | 1-5 | 6-10 |
| 11-14 | 1-5 | 6-当前 |
| 15 | 1-10 | 11-15 |
| 20 | 1-15 | 16-20 |

`k` 表示每次更新完成后的目标原文窗口，不是两次更新之间的绝对上限。

### 12.6 增量更新

每次自动更新时：

- 以当前生效备忘录版本为基础
- 只总结新进入摘要区的轮次
- 不重新总结全部历史
- 用户保护区内容必须原样保留
- 系统摘要区可整理和去重
- 创建新备忘录版本
- 更新摘要覆盖范围

### 12.7 用户编辑

用户保存备忘录修改时：

- 创建新版本
- 保留旧版本
- 版本类型标记为 `USER_EDIT`
- 用户保护区成为系统不可修改内容
- 系统摘要区仍由系统独立维护

### 12.8 版本历史

用户可以：

- 查看全部版本
- 恢复任意历史版本，并创建新的当前版本

用户不能：

- 删除旧版本

恢复规则：

- 不直接修改或重新激活旧版本记录
- 创建 `RESTORE` 类型的新版本
- 记录来源版本和操作前当前版本
- 恢复旧版本的用户保护区和摘要基线
- 对恢复版本未覆盖、且已退出原始窗口的有效历史重新增量总结
- 保留当前最近原始对话

### 12.9 分支继承

在位置 `P` 创建新分支时：

- 只继承摘要覆盖范围严格结束于 `P` 之前的最后一个有效备忘录版本
- 不继承包含分叉消息或其后内容的版本
- 在新分支创建本地继承版本，并记录来源版本
- 如果没有可继承版本，则从无备忘录状态开始
- 分叉后两个分支独立更新

## 16. 验收规则

### 路由

- 每条自动路由消息都必须保存候选评分集合
- 初始选择模型必须是综合路由得分最高的合格模型
- 自动降级顺序必须遵循不可变路由排序
- 手动指定模型时不创建候选评分集合
- 最终实际回答模型必须展示并持久化

### 搜索

- 每个新用户消息节点最多只有一个原始搜索快照
- 所有重新生成操作复用该快照
- 修改历史用户消息后必须创建新搜索快照
- 搜索失败时不得伪造搜索结果

### 回答版本

- 每条用户消息最多一个当前生效成功回答
- 失败的重新生成不得替换当前生效回答
- 已有后续对话时激活早期回答版本必须创建新分支

### 备忘录

- 备忘录必须属于分支
- 用户保护区不得被系统自动修改
- 每次变更必须创建新版本
- 旧版本不可删除
- 新分支不得继承包含分叉点或其后内容的备忘录
- 失败或停止的生成不得触发备忘录更新

### 历史分支

- 修改历史用户消息不得破坏原分支
- 新分支不得包含原分支分叉点之后的消息
- 后续消息和上下文必须由当前活动分支决定

### 错误处理

- 自动模式下，暂时性错误每个模型最多重试一次
- 模型特定确定性错误直接跳过当前模型
- 全局错误立即停止全部降级
- 手动指定模型失败时禁止自动切换
